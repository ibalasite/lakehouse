Local Data Lakehouse
本地資料湖倉

An EDD-compliant, one-click Kubernetes streaming BI pipeline — synthetic data flows every 15 min from a Data Source POD through a Medallion Lakehouse (Bronze→Silver→Gold) to live Metabase dashboards.
EDD 相容一鍵串流 BI pipeline — 合成資料每 15 分鐘從資料來源 POD 流經 Medallion 架構至 Metabase 即時儀表板

platform: k8s k3s Apache Iceberg v2 license: MIT

Quick Start / 快速開始

macOS / Linux / WSL2
bash init_env.sh       # generates .env with fresh random credentials
./k8s/deploy.sh        # deploys the full stack (~90 min on first run — includes 10M-row seed + bulk load)
Windows (PowerShell)
.\init_env.ps1         # generates .env with fresh random credentials
.\k8s\deploy.ps1       # deploys the full stack (~90 min on first run — includes 10M-row seed + bulk load)

Then open http://localhost:30300 and log in with the credentials printed by the init script. The 客服問題單日報 dashboard with 5 charts will be ready.

1. Project Overview 專案概覽

This repository is a self-contained local data lakehouse that demonstrates a production-representative BI pipeline running entirely on a laptop with Rancher Desktop (k3s). It is intended as a reference implementation and learning environment — not a production deployment.

本專案是一個自包含的本地資料湖倉，展示在筆電上以 Rancher Desktop（k3s）執行的生產代表性 BI pipeline，定位為參考實作與學習環境，非生產部署。

Data Source POD→ MinIO/Iceberg→ Polaris→ dbt (Trino)→ MySQL→ Metabase

Pipeline layers / Pipeline 分層

LayerComponentRole

Seed data: the stack generates ~10M rows of synthetic customer support tickets using 04_generate_tickets.py (pyiceberg direct write, bypassing Trino INSERT for speed). After the seed, the Data Source POD continuously generates 5–20 rows every 5 minutes; lakehouse_streaming ingests them every 15 minutes and propagates through the full Medallion pipeline to live Metabase dashboards. The data is purely synthetic — no real user data is included.

2b. Project Goals 專案目標

Goal 1 — Live 15-Minute Streaming Pipeline 15 分鐘串流 Pipeline 全通

The end-to-end pipeline must run and be observable in Metabase without manual intervention:

  1. Data Source POD generates 5–20 synthetic ticket rows every 5 minutes
  2. lakehouse_streaming DAG runs every 15 minutes and completes all tasks:
    ingest_from_source → dbt_bronze → dbt_silver → dbt_hour_long → dbt_hour_wide → cache_daily_report (Iceberg + MySQL) → populate_hourly_cache → verify_dashboard
  3. Metabase 即時監控 auto-refreshes every 15 minutes and shows continuously increasing ticket counts for today

Acceptance: open Metabase, watch cache_daily_report row counts grow every 15 min.

本專案的端對端 pipeline 必須在無人工干預的情況下可在 Metabase 觀測：Data Source POD 每 5 分鐘產生資料，lakehouse_streaming DAG 每 15 分鐘推進 Bronze→Silver→Gold（hour_long→hour_wide）→Cache MV（Iceberg + MySQL 雙目標）→Metabase，即時監控每 15 分鐘自動刷新顯示持續增加的今日問題單數量。

Goal 2 — Credential Output Script 憑證輸出腳本

After deploy.sh completes, all service URLs and credentials must be printed to the console in a structured format separated by access method (WEB UIs / Pod CLI / ClusterIP services).

The script k8s/show_services.sh must also be callable standalone at any time — not just at deploy end — to query current credentials from .env.

bash k8s/show_services.sh   # query current service URLs + credentials at any time

deploy.sh 完畢後必須列印所有服務 URL 與憑證（分 WEB UI / Pod CLI / ClusterIP 存取方式）。k8s/show_services.sh 可隨時獨立呼叫以查詢當前帳號密碼，便於本地開發。

5. Deploy Modes 部署模式

deploy.sh / deploy.ps1 support three operating modes. Run bash init_env.sh (or .\init_env.ps1) before any of them to ensure .env is current.

Default restart / 預設重啟

bash init_env.sh
./k8s/deploy.sh

Reads .env, applies Kubernetes secrets, and redeploys all components. Passwords change on every deploy. PVC data is preserved — existing Iceberg files and MySQL tables remain intact. Use this for routine restarts after a laptop reboot.

Skip seed data / 跳過種子資料

./k8s/deploy.sh --skip-seed

Skips the 04-generate-tickets job. Use this when Iceberg data already exists in the PVC and you only need to redeploy services.

Password rotation (preserve data) / 輪換密碼（保留資料）

./k8s/deploy.sh --rotate

Rotate all credentials and restart stateless services. MySQL and MinIO data are preserved. Internally: regenerates .env, preserves DB root passwords (baked into PVCs), updates the k8s Secret, updates the Trino ConfigMap, runs ALTER USER in MySQL, then rolling-restarts MinIO, Polaris, Trino, Airflow, and Metabase.

Full rebuild / 完全重建

bash init_env.sh        # optional: generate fresh credentials first
./k8s/deploy.sh --rebuild

Tear down all deployments, PVCs, and jobs, then redeploy from scratch. All Iceberg and MySQL data is lost.

8. End-to-End Smoke Test 端對端冒煙測試

e2e_smoke_test.py validates the entire pipeline from data ingestion to Metabase dashboard query. Run it after a fresh deployment or after any change to confirm everything works end-to-end.

python3 platform/init/e2e_smoke_test.py

The test performs these steps in order:

  1. Connectivity check: verifies all 6 services are reachable
  2. Data generation: writes 500 synthetic ticket rows for today into iceberg.bronze.raw_tickets via pyiceberg
  3. Airflow trigger: calls the Airflow REST API to trigger the lakehouse_hourly DAG
  4. DAG poll: polls until the DAG run reaches success or failed (default 600s timeout)
  5. MySQL verify: confirms cache_ticket_hourly contains rows for today's date
  6. Metabase verify: logs in, queries all dashboard card endpoints, confirms rows are returned
  7. Summary: prints PASS or FAIL per step; exits non-zero if any step failed

Options / 選項

python3 platform/init/e2e_smoke_test.py --rows 200       # generate 200 rows instead of 500
python3 platform/init/e2e_smoke_test.py --timeout 900    # allow 900s for Airflow DAG
python3 platform/init/e2e_smoke_test.py --skip-generate  # skip Iceberg write (use existing data)
python3 platform/init/e2e_smoke_test.py --skip-airflow   # run cache refresh locally, skip Airflow

Example output / 範例輸出

[PASS] Connectivity: All services reachable
[PASS] Data generation: 500 rows in 1.3s
[PASS] Airflow DAG (lakehouse_hourly): final state: success
[PASS] MySQL hourly cache: 42 rows for today
[PASS] Metabase dashboard: 318 total rows across cards

All checks passed.

11. Credential Management 憑證管理

How credentials are generated

init_env.sh uses three sources of randomness:

  • openssl rand -hex for usernames and hex tokens
  • /dev/urandom filtered through tr for alphanumeric passwords
  • Python cryptography.fernet.Fernet.generate_key() for the Airflow Fernet key

Every execution of init_env.sh produces a completely different .env. There are no static default passwords anywhere in this codebase.

How credentials reach services

.env
 │
 ├── kubectl create secret generic lakehouse-secrets --from-env-file=.env
 │       │
 │       └── Pod env vars via secretKeyRef in each YAML manifest
 │
 └── deploy.sh envsubst → configmap-trino.yaml (${VAR} replaced at deploy time)
         │
         └── Trino reads catalog config from the substituted ConfigMap

No secret values appear in any committed file. Kubernetes YAML files reference secretKeyRef only.

Python scripts and the _require_env pattern

All Python init scripts use a hard-fail pattern for reading credentials. os.environ.get("KEY", "default") is explicitly banned — if a variable is missing, the script raises an error immediately.

# Correct — raises immediately if var is missing
def _require_env(name: str) -> str:
    value = os.environ.get(name)
    if not value:
        raise RuntimeError(f"Required env var {name!r} is not set. Run: bash init_env.sh")
    return value

MYSQL_PASSWORD = _require_env("MYSQL_PASSWORD")           # correct
MYSQL_PASSWORD = os.environ.get("MYSQL_PASSWORD", "changeme")  # WRONG — never do this

12. Security Design 安全設計

Principle of least privilege / 最小權限原則

MySQL lakehouse user:

  • lakehouse_cache database: SELECT, INSERT, UPDATE, DELETE only — no DDL.
  • metabase database: ALL PRIVILEGES — Metabase requires DDL to manage its own internal schema.

MinIO: Root credentials are used only for bucket creation during the init job. For production, create per-service access keys with bucket-scoped policies.

Apache Polaris: The trino-svc principal is granted all_access_role scoped to the lakehouse catalog only. Vended credentials are disabled (vended-credentials-enabled=false); Trino accesses MinIO directly.

Kubernetes: No hostPath volumes. Credentials injected via secretKeyRef only. No --privileged containers.

Why hardcoded defaults are dangerous

Using os.environ.get("KEY", "hardcoded_default") creates two risks:

  1. Stale credential reuse: if the env var is missing, the script silently uses the hardcoded value instead of failing.
  2. Accidental commit risk: a developer who copies the pattern may commit a file containing a real-looking default that becomes a permanent secret in git history.

No credentials in git history

  • .env is in .gitignore and was never committed
  • All Kubernetes YAML files use secretKeyRef — no literal secret values
  • configmap-trino.yaml in the repo contains only ${VAR} placeholders; real values are substituted at apply time
  • DAG scripts read env vars injected by Kubernetes; no credentials appear in DAG Python source

Production hardening checklist / 生產環境強化清單

  • Enable TLS on all service endpoints
  • Add Kubernetes NetworkPolicies to restrict inter-pod communication
  • Replace MinIO root credentials with per-service IAM-style access keys
  • Integrate a secrets manager (HashiCorp Vault, AWS Secrets Manager, or equivalent)
  • Replace Metabase embedded H2 with a production PostgreSQL backend
  • Add Kubernetes RBAC policies scoped to the lakehouse namespace
  • Enable Polaris persistent storage (replace in-memory catalog with a database backend)
  • Rotate all credentials before first production run

Key file notes

  • platform/data_source/app.py is a pure-stdlib Python HTTP server with enableServiceLinks: false in its pod spec to prevent k8s from injecting DATA_SOURCE_PORT=tcp://... which would crash int() parsing. The pod generates 5–20 rows every 5 minutes into a thread-safe in-memory deque; GET /api/tickets/drain atomically returns and clears the buffer.
  • platform/init/fetch_and_ingest.py polls /api/tickets/drain, converts ISO timestamps to microsecond epoch for PyArrow, and appends to iceberg.bronze.raw_tickets via pyiceberg direct write. Called every 15 minutes by lakehouse_streaming.
  • dbt/models/gold/facts/fact_ticket_hour_long.sql is an append-only EAV narrow fact (one row per grain×field_code). Reads silver via prblm_sysdate > MAX(updated_at) - 1 minute — zero full-table scan, safe for 15-min streaming.
  • dbt/models/gold/facts/fact_ticket_hour_wide.sql is an incremental PIVOT MERGE reading only delta rows from hour_long. 7-dimension composite unique key; trailing 24h watermark.
  • dbt/models/cache/cache_daily_report.sql is the primary Metabase source — Lambda Architecture UNION ALL (D-1 fact_ticket_day_wide + today SUM of fact_ticket_hour_wide) LEFT JOIN all dims. Runs dual-target each cycle.
  • 04_generate_tickets.py writes 10M rows directly to Iceberg via pyiceberg and MinIO, bypassing Trino INSERT. This achieves approximately 200,000–1,000,000 rows per second on typical laptop hardware.
  • populate_mysql_cache.py runs aggregation SQL on Trino and bulk-inserts into MySQL using executemany with 5,000-row batches. The streaming DAG calls it with --streaming which reads from iceberg.gold.fact_ticket_hour_wide for today's date (≤500 rows, ~3s) and writes to cache_ticket_hourly. The daily DAG calls it without flags for a full historical rebuild.
  • configmap-trino.yaml is the only YAML that requires envsubst. deploy.sh substitutes ${VAR} placeholders from .env using an inline Python script before applying.

15. Troubleshooting 疑難排解

Data Source POD CrashLoopBackOff

If the data-source pod crashes with ValueError: invalid literal for int() with base 10: 'tcp://...', Kubernetes injected a DATA_SOURCE_PORT=tcp://ClusterIP:8080 env var overriding the app's own port variable. This was fixed by adding enableServiceLinks: false to the pod spec in k8s/data-source.yaml. Re-running ./k8s/deploy.sh --skip-seed will apply the fix.

Trino pod OOMKilled

Increase the Rancher Desktop VM memory allocation to at least 8 GB (10 GB recommended) in Preferences → Virtual Machine → Memory.

Polaris catalog not found after restart

Polaris uses in-memory persistence by default. After a Polaris pod restart, re-run the bootstrap job:

kubectl -n lakehouse delete job polaris-bootstrap --ignore-not-found
kubectl -n lakehouse apply -f k8s/jobs/02-polaris-bootstrap.yaml
kubectl -n lakehouse wait job/polaris-bootstrap --for=condition=complete --timeout=180s

Trino cannot connect to MinIO

Trino uses the MinIO ClusterIP (not the hostname minio) to avoid Netty DNS resolution failures caused by the k3s ndots:5 search domain. If you modify configmap-trino.yaml, ensure the MinIO endpoint uses the ClusterIP or the full DNS name minio.lakehouse.svc.cluster.local.

configmap-trino.yaml applied with literal ${VAR}

This means kubectl apply -f k8s/configmap-trino.yaml was run directly instead of through deploy.sh. Always apply the Trino ConfigMap via deploy.sh, which performs the envsubst substitution first.

Metabase dashboard shows no data

Verify the MySQL cache tables have been populated, then re-run the setup job:

MYSQL_PW=$(grep ^MYSQL_PASSWORD .env | cut -d= -f2)
kubectl -n lakehouse exec deployment/mysql -- \
  mysql -ulakehouse -p"${MYSQL_PW}" \
  -e "SELECT COUNT(*), MAX(date_sk) FROM lakehouse_cache.cache_ticket_daily;"

kubectl -n lakehouse delete job metabase-setup --ignore-not-found
kubectl -n lakehouse apply -f k8s/jobs/05-metabase-setup.yaml

e2e_smoke_test.py fails at Airflow step

Ensure the Airflow scheduler has loaded the DAGs. If the scheduler was recently restarted, wait 30 seconds for DAG discovery:

kubectl -n lakehouse rollout restart deployment/airflow-scheduler
# wait ~30 seconds, then retry
python3 platform/init/e2e_smoke_test.py --skip-airflow   # local fallback
