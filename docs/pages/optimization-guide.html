為什麼 MSTR 報表在 Athena 跑不出來？

先理解根本原因，才能對症下藥

背景知識：MicroStrategy Group 功能 MSTR 的「Group By Element」或「Custom Group」功能，會把每一個群組條件拆成獨立的 SQL 語句，再由 MSTR 伺服器把結果合併。這份報表有 77 個群組，就產生了 77 條 SQL，每條都去掃同一批 View。

確認 Athena MV 的技術限制

先確認哪些路走不通，避免白工

背景知識：Athena Iceberg Materialized View AWS 官方文件明確寫道："Athena does not support CREATE MATERIALIZED VIEW, REFRESH MATERIALIZED VIEW... To create materialized views, use Apache Spark in Amazon EMR or AWS Glue."（來源：docs.aws.amazon.com/athena/latest/ug/querying-iceberg-gdc-mv.html）

dbt-athena adapter 也沒有 MV materialization，只有待實作的 Issue #1557。

✅ 結論使用 dbt incremental + Iceberg table，不需要 Glue MV，不需要 Spark Job，就能達到「增量更新 + partition 加速」。

繪製現有 View Chain 全圖

在動手之前，必須知道每一層的依賴關係

背景知識：Athena View vs Iceberg Table Athena 的 VIEW 是「查詢定義的別名」，每次查詢都即時展開執行，沒有任何快取或 partition 效果。Iceberg Table 是真實儲存在 S3 的 Parquet 檔案，支援 partition pruning（只讀目標分區）、time travel、ACID merge。

理解 dbt Materialization 策略

選對 materialization，才能做到增量更新不全表重算

背景知識：dbt 的四種 Materialization dbt 提供四種方式將 model 寫入資料庫，針對這個場景，incremental + Iceberg merge 是唯一能做到「不 full refresh、有 partition、有 ACID」的方案。

背景知識：Iceberg Incremental Strategy — merge dbt-athena 的 incremental 支援 merge 策略：根據 unique_key 比對，有則 UPDATE，無則 INSERT。底層使用 Iceberg MERGE INTO 語法，只修改有變動的檔案（copy-on-write 或 merge-on-read），不重寫整張表。

建立 dbt Incremental Iceberg Model

這是核心步驟：把 View 轉成有 partition 的物理表

背景知識：Iceberg Partition Transform Iceberg 支援 day(欄位) / month(欄位) / year(欄位) 等 hidden partition，查詢時 Athena 自動 prune（跳過不相關分區）。MSTR 的每條 SQL WHERE 都有日期範圍（如 prblm_sysdate >= '2025-06-21'），partition 後只掃目標日期的資料夾。

-- models/gold/mv_case_handling_detail.sql
{{ config(
    materialized='incremental',
    table_type='iceberg',
    incremental_strategy='merge',
    unique_key='prblm_code',
    partitioned_by=['day(prblm_sysdate)'],
    format='parquet',
    on_schema_change='sync_all_columns'
) }}

SELECT
  fpm.prblm_code
, fpm.prblm_sysdate
, fpm.prblm_blngcode                               AS "RM單號"
, fpm.usr_id, fpm.gd_id, fpm.catsub_id
, fpm.prblm_processuser
, fpm.prblm_preassignenddate
, fpm.prblm_statusdate, fpm.prblm_donedate
, fpm.prblm_doneatatime, fpm.prblm_doneuser
, fpm.prblm_status_id, fpm.prblm_complain_id
, fpm.prblm_perform_id, fpm.prblm_prtype_id
, fpm.prblm_source_id, fpm.prblm_intclass_id
, fpm.prblm_class_id, fpm.supplier_id
, fpm.prblm_notasreason_id, fpm.prblm_notasowndept_id
-- 預計算 DATE_DIFF（不再讓每條 MSTR query 即時算）
, DATE_DIFF('hour', fpm.prblm_sysdate, fpm.prblm_donedate)           AS "結案時效"
, DATE_DIFF('hour', fpm.prblm_sysdate, fpm.prblm_preassignenddate)   AS "回覆時效"
, CASE
    WHEN DATE_DIFF('hour', fpm.prblm_sysdate, fpm.prblm_preassignenddate) <= 4   THEN '4hr內回覆'
    WHEN DATE_DIFF('hour', fpm.prblm_sysdate, fpm.prblm_preassignenddate) <= 8   THEN '4hr~8hr回覆'
    WHEN DATE_DIFF('hour', fpm.prblm_sysdate, fpm.prblm_preassignenddate) <= 24  THEN '8hr~24hr回覆'
    ELSE '24hr以上回覆' END                                                  AS "回覆時效群組"
-- 所有 dim join（跟原 view 相同）
, v1."子站名稱", v1."區域名稱", v1."區域編號"
, dpfq.Quename AS "客服大分類名稱", dpfq.Quek_id AS "客服大分類編號"
, dsr.supplier_storeid, dsr.supplier_storename, dsr.supplier_name
-- ... （其餘欄位照 mv_case_handling_detail view 完整複製）
FROM prod_sale_gold.fact_prblm_m fpm
LEFT JOIN prod_sale_gold.fact_answr a      ON fpm.prblm_code = a.answr_prblmcode
LEFT JOIN prod_sale_gold.mv_product_line_unit v1 ON fpm.catsub_id = v1."子站編號"
LEFT JOIN prod_sale_gold.dim_prblm_faq dpfq  ON fpm.prblm_intclass_id = dpfq.Queid
LEFT JOIN prod_sale_gold.dim_supplier dsr    ON fpm.supplier_id = dsr.supplier_id
-- ... 其餘 13 個 LEFT JOIN 同 view 定義

{% if is_incremental() %}
WHERE fpm.prblm_sysdate >= date_add('day', -3, current_timestamp)
-- 往前 3 天：涵蓋補單、狀態更新的資料
{% endif %}

首次執行 Full Load

第一次 run 需要建立完整的歷史資料

背景知識：--full-refresh flag dbt incremental model 首次執行時，is_incremental() 回傳 False，所以會全量建表。若之後要強制全表重建（例如 schema 改變），用 --full-refresh flag。正常日常 run 不加這個 flag，只跑增量。 
# 首次全量建表（只做一次）
dbt run --select mv_case_handling_detail --full-refresh

# 驗證資料筆數與原 view 一致
# 在 Athena 查詢：
SELECT COUNT(*) FROM prod_sale_gold.mv_case_handling_detail;
SELECT COUNT(*) FROM <原本的 view>;
⚠ 注意 首次 full refresh 會掃所有歷史資料，費用較高，屬一次性成本。之後每日增量 run 只掃 3 天，成本大幅下降。

把舊 View 指向新 Iceberg Table

讓 MSTR SQL 完全不需要修改

背景知識：為什麼不直接刪 View？ MSTR 的 SQL 寫死了 "mv_case_handling_detail" 這個名稱，不能改。如果 dbt model 建在不同 schema，就需要讓 view 變成一個「thin wrapper」指向新表。如果 dbt model 建在同一 schema（推薦），則可以直接 DROP VIEW，讓 dbt table 使用同名。
flowchart LR subgraph "方案 A：同 Schema（推薦）" A1["dbt model 建表\nprod_sale_gold.mv_case_handling_detail\n(Iceberg Table)"] A2["DROP VIEW prod_sale_gold.mv_case_handling_detail\n（在 dbt run 前執行，或由 dbt 自動 replace）"] end subgraph "方案 B：不同 Schema" B1["dbt model 建表\nanalytics.mv_case_handling_detail_iceberg"] B2["修改原 VIEW 為：\nSELECT * FROM analytics.mv_case_handling_detail_iceberg"] end style A1 fill:#dcfce7,stroke:#16a34a,color:#000 style B2 fill:#fef9c3,stroke:#ca8a04,color:#000 
-- 方案 A：先在 Athena 刪掉舊 view（dbt run 會自動建同名 table）
DROP VIEW IF EXISTS prod_sale_gold.mv_case_handling_detail;

-- 執行 dbt（此時 is_incremental() = false，全量建表）
-- dbt 會在 prod_sale_gold 建立同名 Iceberg Table
dbt run --select mv_case_handling_detail

-- 方案 B：保留 view 作 thin wrapper
CREATE OR REPLACE VIEW prod_sale_gold.mv_case_handling_detail AS
SELECT * FROM analytics.mv_case_handling_detail_iceberg;

Airflow DAG 設計：Athena 先、MySQL 後

你的架構是 Airflow + dbt → Athena/Iceberg → MySQL，DAG 順序至關重要

背景知識：為什麼 MySQL 必須在 Athena 之後？ MySQL 的 2 年熱資料是從 Athena（或同一批 source 系統）同步而來。Airflow DAG 需要確保 Athena 的 dbt run 完成（Iceberg table 已更新）後，才觸發 MySQL 的 dbt run，避免 MySQL 讀到舊資料。兩個 dbt run 共用同一份 model SQL，只是 target 不同（--target athena vs --target mysql）。

gantt title Airflow DAG 每日執行時序（UTC+8） dateFormat HH:mm section 上游 ETL Source → Athena fact tables :done, 01:00, 2h Source → MySQL fact tables :done, 02:00, 1h section Athena dbt dbt run --target athena :active, 05:00, 30m dbt test --target athena :05:30, 10m section MySQL dbt（等 Athena 完成） dbt run --target mysql :05:40, 15m dbt test --target mysql :05:55, 5m section MSTR 報表 使用者查詢 :crit, 07:00, 4h 
# airflow/dags/mv_case_handling_detail_refresh.py
from airflow import DAG
from airflow.operators.bash import BashOperator
from datetime import datetime, timedelta

with DAG(
    dag_id="mv_case_handling_detail_refresh",
    schedule_interval="0 21 * * *",  # UTC 21:00 = TWN 05:00
    start_date=datetime(2026, 1, 1),
    catchup=False,
) as dag:

    # ① Athena/Iceberg（先跑）
    athena_run = BashOperator(
        task_id="dbt_athena_run",
        bash_command="dbt run --select mv_case_handling_detail --target athena",
    )
    athena_test = BashOperator(
        task_id="dbt_athena_test",
        bash_command="dbt test --select mv_case_handling_detail --target athena",
    )

    # ② MySQL（等 Athena 完成後才跑）
    mysql_run = BashOperator(
        task_id="dbt_mysql_run",
        bash_command="dbt run --select mv_case_handling_detail --target mysql",
    )
    mysql_test = BashOperator(
        task_id="dbt_mysql_test",
        bash_command="dbt test --select mv_case_handling_detail --target mysql",
    )

    # DAG 依賴：Athena 完成後才跑 MySQL
    athena_run >> athena_test >> mysql_run >> mysql_test

同樣的思路，不同的工具選擇

建立 dbt MySQL Incremental Model

因為資料量小，可先用 table materialization，需要增量再升級

背景知識：千萬筆資料為何必須用 incremental 2 年歷史 × 千萬筆若每次全量重建，MySQL 需要 10–30 分鐘，遠超 MSTR 報表前的時間窗口。incremental + delete+insert 每次只處理最近 3 天的資料（約數萬筆），幾十秒就完成。兩套環境統一用 incremental，維護方式也完全一致。

-- models/gold/mv_case_handling_detail.sql（MySQL 版）
{{ config(
    materialized='incremental',
    incremental_strategy='delete+insert',
    unique_key='prblm_code'
) }}

SELECT
  fpm.prblm_code
, fpm.prblm_sysdate
, fpm.prblm_blngcode
, TIMESTAMPDIFF(HOUR, fpm.prblm_sysdate, fpm.prblm_donedate)          AS `結案時效`
, TIMESTAMPDIFF(HOUR, fpm.prblm_sysdate, fpm.prblm_preassignenddate)   AS `回覆時效`
, CASE
    WHEN TIMESTAMPDIFF(HOUR, fpm.prblm_sysdate, fpm.prblm_preassignenddate) <= 4   THEN '4hr內回覆'
    WHEN TIMESTAMPDIFF(HOUR, fpm.prblm_sysdate, fpm.prblm_preassignenddate) <= 8   THEN '4hr~8hr回覆'
    WHEN TIMESTAMPDIFF(HOUR, fpm.prblm_sysdate, fpm.prblm_preassignenddate) <= 24  THEN '8hr~24hr回覆'
    ELSE '24hr以上回覆' END                                               AS `回覆時效群組`
, v1.catzone_name AS `區域名稱`
, v1.catzone_id   AS `區域編號`
, dsr.supplier_storeid, dsr.supplier_storename, dsr.supplier_name
-- ... 其餘欄位同原 view 完整複製
FROM fact_prblm_m fpm
LEFT JOIN fact_answr a       ON fpm.prblm_code = a.answr_prblmcode
LEFT JOIN dim_catzone v1     ON fpm.catzone_id = v1.catzone_id
LEFT JOIN dim_prblm_faq dpfq ON fpm.prblm_intclass_id = dpfq.Queid
LEFT JOIN dim_supplier dsr   ON fpm.supplier_id = dsr.supplier_id
-- ... 其餘 LEFT JOIN

{% if is_incremental() %}
WHERE fpm.prblm_sysdate >= DATE_SUB(NOW(), INTERVAL 3 DAY)
-- 涵蓋補單與狀態延遲更新
{% endif %}
M2

加上 Index 取代 Iceberg Partition Pruning

MySQL 沒有 hidden partition transform，要手動建 index

背景知識：MySQL B-tree Index vs Iceberg Partition Iceberg 的 day(欄位) 是檔案層級的實體分區，Athena 直接跳過不相關 S3 資料夾。MySQL 的 B-tree index 掃到目標 leaf node 後直接讀取 row pointer，效果接近但機制不同。針對 MSTR 的日期範圍查詢，在 prblm_sysdate 上加 index 是最有效的手段。 
-- dbt run 建表後，立即在 Athena Query Editor 執行：
-- （或在 dbt post-hook 裡加）

ALTER TABLE mv_case_handling_detail
  ADD INDEX idx_sysdate       (prblm_sysdate),
  ADD INDEX idx_status_date   (prblm_status_id, prblm_sysdate),
  ADD INDEX idx_catsub        (catsub_id),
  ADD INDEX idx_supplier      (supplier_id);

-- 驗證 Index 有被使用：
EXPLAIN
SELECT * FROM mv_case_handling_detail
WHERE prblm_sysdate BETWEEN '2025-06-21' AND '2025-06-26'
  AND prblm_prtype_id = 1;
-- 確認 key 欄位顯示 idx_sysdate，type 為 range
graph LR subgraph "無 Index（舊 View）" Q1["WHERE prblm_sysdate\nBETWEEN 06/21 AND 06/26"] --> FULL["Full Table Scan\n掃全部 2 年資料"] end subgraph "有 Index（物理表）" Q2["WHERE prblm_sysdate\nBETWEEN 06/21 AND 06/26"] --> IDX["B-tree Index Scan\n直接定位目標 leaf"] IDX --> ROW["只讀目標 5 天資料"] end style FULL fill:#fee2e2,stroke:#ef4444,color:#000 style ROW fill:#dcfce7,stroke:#16a34a,color:#000
dbt post-hook 自動建 Index 可以在 config 裡加 post-hook，讓 dbt run 後自動執行：
post_hook=["CREATE INDEX IF NOT EXISTS idx_sysdate ON {{ this }} (prblm_sysdate)"]

替換 View 並設定排程

MySQL 的 View 替換方式與 Athena 相同

-- 1. 刪掉舊 view（dbt run 會建同名 table）
DROP VIEW IF EXISTS mv_case_handling_detail;

-- 2. 首次全量建表
dbt run --select mv_case_handling_detail --full-refresh

-- 3. 驗證資料筆數
SELECT COUNT(*), MIN(prblm_sysdate), MAX(prblm_sysdate)
FROM mv_case_handling_detail;

-- 4. 後續每日排程（crontab 或 Airflow）
-- 05:00 每天執行：
-- dbt run --select mv_case_handling_detail
